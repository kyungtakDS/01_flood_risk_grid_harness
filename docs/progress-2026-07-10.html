03결정사항

서브 에이전트 모드로 설계

대시보드 생성은 단일 전문가 작업(실시간 협업 불필요)이라 에이전트 팀 대신 서브 에이전트 모드를 선택. 팀 통신 오버헤드를 피함.

도메인 카운트 8/9 유지(도구는 제외)

대시보드 수치는 홍수위험 도메인 하네스 기준(에이전트 8·스킬 9). 새로 추가한 dashboard-builder/htmlity는 "만드는 도구"라 카운트에서 제외 — README 배지·config와 일치 유지. 규칙은 data-extraction.md에 기록.

기존 dashboard.html 보존

사용자 요청대로 내용 무변경. git에는 add만 하여 원본을 그대로 포함, 새 대시보드는 dashboard_v2.html로 분리.

main 직접 푸시

기본 지침은 "기본 브랜치면 새 브랜치"이나, 이 저장소는 개인 소유·이력 전부 main·명시적 push 요청이라 관행에 맞춰 main에 커밋·푸시.

htmlity ↔ dashboard-builder 경계

htmlity는 서술형·구조화 진행 문서(순수 HTML/CSS, 외부 의존 0), dashboard-builder는 지표·Chart.js 대시보드. description에 상호 구분을 명시해 트리거 충돌을 방지.

04검증

  • 대시보드 구조 — 캔버스 6 ↔ doughnut() 6 ↔ legend() 6 일치, 외부 <script>는 Chart.js CDN 1개뿐, 섹션 태그 7/7 균형.
  • 대시보드 렌더 — 크롬에서 전 구간 스크린샷 확인, 도넛 6개·이미지·타임라인 정상, 콘솔 에러 없음.
  • dashboard.html 무변경 — git에서 내용 변경 없이 신규 add만 확인.
  • 푸시 동기화 — git status가 main ↔ origin/main 동기화(ahead/behind 없음) 확인.
  • htmlity 문서 — 자체완결(외부 의존 0), TOC 앵커 동작, 표시 사실이 실제 파일·커밋과 일치.
